# תוכנית מעבר: BASE44 → Supabase + Vercel

**תאריך:** 2026-06-25
**מערכת:** Legal Flow CRM (משרד עו"ד אוהד טבת)
**מטרה:** אותה מערכת ואותו UI בדיוק, backend עובר ל-Supabase (DB + auth + storage +
edge functions) ו-Vercel (אירוח). תוצאה: סוף לתשלום ל-BASE44, שליטה מלאה, הכל ב-git.
**סיבת המעבר:** עלות + מגבלות/שליטה.

---

## נתוני האודיט (2026-06-25)

מספרים מהקוד החי (`github.com/ohad-law/legalflow-crm`):
- 57 מסכים · 131 קומפוננטים · 43 פונקציות ענן · 31 טבלאות · 268 קריאות base44 SDK · 10+ אינטגרציות.

  לידים שת"פ יניב, צ'אט פנימי, חשבוניות, דשבורד מע"מ.

  SocialMediaAccount, SocialMediaPost, WebsiteAnalytics, InternalMessage, LeadTalush,
  LeadShatafYaniv, Invoice, Expense.
- **פונקציות (~12):** fetchNetHamishpatEmails, netHamishpatWebhook, generateViralScript,
  fetchTikTokStats, submitTalushLead, notifyTalushLead, talushSheetsWebhook,
  initiateTalushBotConversation, shatafYanivWebhook, internalChat, fetchInvoicesFromEmail,
  scanExpense. (כוללות את האינטגרציות הקשות ביותר — נט המשפט/OTP, TikTok, Gmail.)

### מה נשאר (ליבה)
- **טבלאות (~18):** Case, Client, Lead, Task, Attendance, WorkLog, CaseMilestone,
  SystemSettings, User, UserPermission, SignatureRequest, SignatureField, Income,
  MarketingCampaign, WorkerIntakeForm, Folder, CaseDocument, Message, NotificationLog.
- **שיווק + לידים מפייסבוק נשארים** (אוהד מריץ Meta ads).

### החלקים הקשים (סיכונים)
1. 3 חיבורי Google OAuth שמנוהלים היום ע"י BASE44 (Drive, Gmail, יומן) — צריך אפליקציית
   Google משלנו + אחסון/רענון טוקנים.
2. Auth + הרשאות שזורים ב-28 מתוך 57 מסכים → Supabase Auth + טבלת roles + RLS + shim ל-User.me().
3. 268 קריאות SDK → שכבת תרגום (db.Case.list() וכו') מעל Supabase כדי שהשינויים בפרונט מינימליים.
4. Webhooks (Facebook, Make, ManyChat, Green API, sheets) → כתובות חדשות ורישום מחדש בלוחות
   הבקרה החיצוניים (דורש את אוהד).
5. סודות להעביר: Facebook (4), Gmail (2), Green API (2), Finbot (1).
6. לאחד לוגיקת המרה כפולה (`onLeadConvertedToClient` מול הפיצ'ר בפרונט).

### נתון חסר
נפח המידע החי (כמה לקוחות/תיקים/לידים/משימות) — אין לאוהד מספרים. **יימשך בשלב 0 דרך טוקן
BASE44 / בדיקה בדאשבורד.** קובע את אורך שלב 4.

## השלבים

### שלב 0 — הכנה (לא נוגעים במערכת החיה)
הקמת Supabase + Vercel, פתיחת אפליקציית Google משלנו, גיבוי מלא של נתוני BASE44,
ומשיכת נפח המידע.
- **מאוהד:** גישה לפתיחת חשבונות (חינמיים), אישור גיבוי.
- **סיכון:** אפס. · **זמן:** 2-3 ימים.

### שלב 1 — מסד נתונים + התחברות (השלד)
18 טבלאות ב-Supabase + relations + RLS, מערכת התחברות והרשאות, ושכבת התרגום.
- **מאוהד:** כלום. · **סיכון:** אפס (החי ממשיך לרוץ). · **זמן:** 3-5 ימים.

### שלב 2 — המסכים
הפניית ~47 המסכים לשכבת התרגום, העלאה ל-Vercel, אותו UI, בדיקה על נתוני דמה.
- **מאוהד:** לאשר שזה נראה זהה. · **סיכון:** נמוך. · **זמן:** 4-6 ימים.

### שלב 3 — הפונקציות והאינטגרציות (החלק הארוך)
העברת ~25 פונקציות לפי סדר חשיבות: ליבה (וואטסאפ, Drive, חתימות, מסמכים) → webhooks
(לידים פייסבוק, Make) → מתוזמנות (התראות דיונים). איחוד לוגיקת ההמרה הכפולה.
- **מאוהד:** חיבור מחדש של Google/WhatsApp + עדכון כתובות webhook. **צוואר הבקבוק האמיתי.**
- **סיכון:** בינוני. · **זמן:** 1-2 שבועות (תלוי בזמינות אוהד).

### שלב 4 — העברת המידע החי + הרצה במקביל
ייצוא נתונים מ-BASE44 → המרה → טעינה ל-Supabase. שתי מערכות במקביל, וידוא תאימות.
- **מאוהד:** לאשר שהמידע נכון. · **סיכון:** הכי גבוה — לכן אחרון, עם גיבוי. · **זמן:** 3-5 ימים.

### שלב 5 — מעבר סופי
הפניית דומיין ל-Vercel, מעקב כמה ימים, BASE44 כגיבוי קריאה-בלבד, ואז **ביטול המנוי**.
- **מאוהד:** אישור סופי. · **סיכון:** נמוך (יש דרך חזרה עד הביטול). · **זמן:** 2-3 ימים מעקב.
